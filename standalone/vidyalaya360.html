Executive summary

What it is, who it serves, and why it exists.

Vidyalaya360 is a single web console that replaces the half-dozen disconnected spreadsheets, WhatsApp groups and paper registers a typical Indian K-12 school runs on today. One application; four roles (Super Admin / Principal / Teacher / Parent); eighteen screens; one source of truth backed by a real Postgres database.

The brief was to make every action a school's office actually performs in a day — taking an admission, collecting a fee, logging a daily class report, marking a bus rider absent, raising a complaint — possible from a single screen, with a full audit trail and zero data loss. That's what this report describes.

Technology stack

A small, modern stack chosen for ergonomics, not novelty.

Frontend

Next.js 14 App Router with React 18 Server & Client Components. Styled with hand-written CSS variables (no Tailwind, no UI kit). Fonts: DM Serif Display for editorial headings, Outfit for UI, JetBrains Mono for numerics.

Next.js 14.2 React 18.3 CSS variables Google Fonts

Backend

Server-side API routes (one per resource) talking to Supabase via the official JS SDK. The server uses the service_role key to bypass Row-Level Security; the anon key is never used for writes.

Next.js API routes @supabase/supabase-js Service-role auth

Database

Supabase-hosted Postgres. 9 tables, RLS enabled on every table, idempotent schema (every CREATE uses if not exists), JSONB column for nested route stops, composite primary keys for the daily-log upsert.

Supabase Postgres 15 Row-Level Security JSONB

Persistence fallback

When the three Supabase env vars are absent the same async API silently falls back to a local JSON file at data/db.json — useful for offline demos and local dev with zero setup.

Optional JSON file store Same async API

Every mutation flows server-side through one of the API routes. The client never talks to Supabase directly — that's how the secret service-role key stays out of the browser. After every mutation the client calls refresh(), which re-fetches /api/data, so the UI reflects DB state on every interaction.

Features in detail

Every screen, every interactive element, every workflow.

Cross-cutting features (work on every screen)

These are wired into the application shell, so they're available regardless of which screen you're on.
  • Sidebar navigation — role-based menu. The list of items changes when you switch role; sections are grouped (Trust, People, Operations, CRM, System). Click any item to switch screens; the active row gets an amber left-border + soft fill.
  • Sidebar collapse / expand — toggle via the menu button in the topbar or the Tweaks panel. Collapsed mode shows icons only (58 px wide).
  • Topbar search — placeholder copy adapts to the current role ("Search students, fees, staff, routes…" for Principal vs "Search fees, messages, transport…" for Parent). Keyboard shortcut hint: ⌘F.
  • Live pulse-dot — green pill in the topbar that pulses to indicate the system is in real-time mode. Hidden when role is Parent.
  • Theme toggle — sun / moon icon in the topbar flips the entire colour palette (light cream ↔ deep ink), persists to localStorage.
  • Notifications icon — small dot indicator; opens a notification tray (placeholder).
  • Tweaks panel (⌘K) — slide-in side panel with Role, View (Desktop/Mobile), Theme, Density (compact/balanced/spacious), Sidebar (expanded/icons), Accent colour swatches.
  • Role pill — floating dark pill at the bottom-centre that swaps roles instantly (Super Admin / Principal / Teacher / Parent). Triggers a re-render of the sidebar + landing screen.
  • Desktop / Mobile view toggle — bottom-right floating pill that wraps the whole app in an iPhone frame (390 × 844) for mobile-responsive previewing.
  • Toast notifications — top-centre rounded pill (green = ok, red = bad, amber = warn) that confirms every mutation: "Admitted Aakash Kummar · STN-9164", "Withdrew Aakash · history preserved", "5 reminders queued", etc. Auto-dismisses after 2.4 seconds.

Money Control

Principal · Super Admin

Real-time financial picture — fees in, expenses out, what's pending.

  • Four live KPIs computed from real Supabase data:
    • Total income · YTD = sum of recent_fees.amount
    • Total expense · YTD = ₹0 ("not tracked yet" — explicit honesty rather than fake)
    • Net surplus = income − expense, with margin %
    • Pending receivables = sum of pending_fees.amount + count of students
  • Combined / School only / Trust only segmented filter.
  • Income vs Expense chart (12-week area + bar) — empty-state placeholder when no series exists.
  • Income breakup card — four bars (Fees received, Fees pending, Donations, Other income) with width proportional to share, ₹K + percent display.
  • Ledger table — every recent_fee rendered as a transaction row with TXN id, date, description, account chip (School / Trust), category, method, ± amount (green income / red expense), Receipt button.
  • Filter tabs — All / Income / Expense.
  • Export ledger button.
Derived view — no direct writes. Reflects every POST /api/fees/pay in real time.

Fees & UPI

Principal · Parent

The most-used screen. End-to-end fee collection, from picking a student to issuing a printable receipt.

  • Live total chips — ₹X collected (green), ₹Y pending (warn), ₹Z overdue (red), all computed from current data.
  • Header buttons:
    • Import structure — opens an OS file picker (.csv / .xlsx); on selection, toast confirms file name + size.
    • Export CSV — builds a CSV from the current data and triggers a Blob download (filename includes ISO date).
    • Collect fee · N pending — opens a quick-collect dropdown listing every pending student (avatar, name, class, ID, due date, amount in red if overdue). Click one to load it into the right panel.
  • Status segmented filter — All · N / Paid · N / Pending · N / Overdue · N. Counts are live.
  • Class filter button — opens a dropdown menu of classes 1–8.
  • Bulk select checkboxes — header checkbox selects all unpaid in the current view; per-row checkboxes for individual selection (paid rows disabled). When ≥1 row is selected, an amber bar appears with Remind on WhatsApp / Remind via SMS buttons that POST to /api/fees/remind — composite activity-feed entry plus one audit row per recipient.
  • Per-row Collect button (pending rows) — loads that student into the right panel.
  • Per-row Receipt button (paid rows) — jumps the right panel to Step 3 with that fee selected.
  • Right panel: 3-step UPI flow
    • Step 1 (Pick) — student card with avatar, name, ID, parent phone, line items (Term fee, Activity fee, Discount, Late fee), Payable total in serif font. Method selector (UPI / Cash / Bank). Proceed button.
    • Step 2 (QR) — fake but scannable-looking QR (deterministic SVG pattern), UPI handle "stansford@hdfc", "Listening for payment…" pulse pill. Buttons: Back (return to Step 1) and Mark paid (calls POST /api/fees/pay).
    • Step 3 (Receipt) — printable monospace receipt with header "STANSFORD INTERNATIONAL HR.SEC.SCHOOL", student details, line items, PAID stamp. Four share buttons (WhatsApp / SMS / PDF / Email) — see below. Done · collect another button auto-advances to the next pending student.
  • Stage indicator dots at the top of the panel are clickable shortcuts when valid.
  • Receipt share actions:
    • WhatsApp opens wa.me/<parent-phone> with prefilled receipt text.
    • SMS opens sms: link.
    • PDF calls window.print() — uses the browser's save-as-PDF.
    • Email opens mailto: with subject + body prefilled.
Writes to recent_fees, pending_fees, activities, audit_log, and the matching students.fee field.

Audit log

Super Admin

Immutable trail of every server-side action.

  • Four computed KPIs — Events total, Financial writes (matches "paid"/"receipt"/"reminder"/"donation"), Permission changes (matches "role"/"user"), Failed attempts.
  • Streaming chip in the card head (teal pulse).
  • Activity list — per-row time (mono), avatar, action title, who + meta sub-line, tone chip. The tone is computed via toneFor(action):
    • ok — paid, receipt, board, resolved, converted
    • bad — absent, complaint
    • warn — progress, contacted, late
    • info — created, enquiry, viewed
    • accent — anything else
  • Filter dropdown: All events / Financial / Permissions / Automations.
  • Export button.
Append-only — never edited or deleted. Receives one row per mutation across every API route.

Production rule: never lose records

Soft delete only. The Withdraw action on a student flips status='archived' and stamps archived_at; their paid receipts (recent_fees), daily logs (daily_logs) and every audit row are kept forever. Only the uncollected pending fee is dropped because we no longer expect to collect it. Restore reverses the flag and re-creates a fresh pending fee for the current term.

Engineering decisions worth noting

A handful of choices that defined the build.

1 · Pluggable persistence behind one async API

lib/db.js exposes a single async API (readAllData, addStudent, payPendingFee, …) and branches internally on whether the Supabase env vars are present. The same codebase runs against real Postgres in production and a local JSON file in dev — zero changes to API routes or screens.

2 · Soft delete, not cascade delete

An earlier version cascade-deleted a student's fees and logs alongside the student row. That violated the basic accounting rule that paid receipts and academic records must be permanent. The current Withdraw action only flips a status flag; financial and academic history are kept forever, with a Restore path back.

3 · KPIs computed from real data, not seeded numbers

The Dashboard and Money screens initially read from a static KPIS seed that was zeroed when mock data was removed — leaving "₹0 collected" cards even after a real ₹17,000 receipt. Fixed by replacing static reads with reduce calls over the live tables: collected = sum of recent_fees, pending = sum of pending_fees, etc.

4 · Hydration mismatch from clock-dependent SSR

Two values rendered new Date() directly inside JSX — the Dashboard greeting/date and the Academic week-of dropdown. Server (UTC) and client (IST) could disagree across a day boundary, producing the "Text content does not match server-rendered HTML" warning. Both are now computed in a useEffect after mount, with a stable placeholder for the first paint.

5 · Floating menus that escape table clipping

The student row's "..." menu was being cropped to one item because the table wrapper sets overflow-x: auto — which, by browser rule, also forces overflow-y to clip. Switched the menu to position: fixed anchored to the trigger button's bounding rect, with auto-flip when it would fall off the bottom of the viewport.

6 · Network failures become toasts, not error overlays

Every fetch in the Students screen now goes through a safeFetch() wrapper that catches network rejections and returns a {ok: false, json.error} shape. If the dev server hot-reloads mid-click, you get a friendly red toast instead of a whole-page error overlay.

7 · Per-table error swallowing in readAllData

Each Supabase select goes through a safeSelect helper that catches per-table errors. If a single table is missing (e.g. you haven't run schema.sql), the response still returns 200 with an empty array for that table and a single [db] <table>: <msg> warning lands in the server console — making the missing-migration case trivially diagnosable.

8 · Indian mobile validation, normalised at the boundary

New admissions accept 10 digits starting with 6/7/8/9. A courtesy +91 / 91 / 0 prefix gets stripped before counting digits, and the stored value is always the canonical +91 XXXXX XXXXX. Client-side and server-side enforce the same rule — client for instant feedback, server as the authority.

How to run

Two paths — local file backend for offline dev, Supabase for production.

Local (zero config)

git clone https://github.com/aakashkummar-workspace/Stansford_crm.git
cd Stansford_crm
npm install
npm run dev
# open http://localhost:3000

Without env vars the app uses a JSON file at data/db.json. Delete that file to reset.

With Supabase (production)

# 1. Create a project at https://supabase.com
# 2. SQL editor → paste lib/schema.sql → Run
# 3. Copy .env.example → .env.local and fill:
NEXT_PUBLIC_SUPABASE_URL=https://YOURPROJECT.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJ...        # public, safe in browser
SUPABASE_SERVICE_ROLE_KEY=sb_secret_...     # SECRET — server only

# 4. Run it
npm run dev
curl -i http://localhost:3000/api/data | grep x-data-backend
# x-data-backend: supabase

Where it can go next

Things deliberately left for v2.

Auth

Currently single-tenant — anyone with the URL gets in. Wire up Supabase Auth so each role maps to a real user, then turn on RLS policies for client-side reads.

Real expense tracking

The Money screen shows expense as ₹0 ("not tracked yet") — add an expenses table with categories (payroll, utilities, etc.) and an entry screen.

Notifications

The WhatsApp / SMS reminder buttons currently log audit rows but don't actually send. Plug in MSG91 or Twilio for real delivery.

Multi-school trust

Add a school_id foreign key on every entity and filter by it. The schema already uses a schools concept on the Super Admin trust screen.
